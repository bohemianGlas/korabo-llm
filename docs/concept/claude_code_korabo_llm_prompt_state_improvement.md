# Claude Code向け改修指示書  
## korabo_llm：マルチエージェント執筆における責務分離・状態管理・プロンプト改善

あなたは、既存の `korabo_llm` リポジトリを改修するシニアソフトウェアアーキテクト兼LLMアプリケーションエンジニアです。

この指示書の目的は、Master LLMとキャラクター・ロールLLMの責務を明確に分離し、長編執筆時の整合性、人物の主体性、情報隔離、外部記憶の品質を改善することです。

最初にリポジトリ全体を調査し、既存の設計、データモデル、プロンプト生成箇所、LLM呼び出し経路、履歴管理、記憶更新、プリセット入出力、テスト構成を把握してください。そのうえで、既存機能と後方互換性を可能な限り維持しながら改修してください。

質問で作業を止めず、不明点には妥当な仮定を置き、その仮定を最終報告に明記してください。

---

# 1. 現在のシステム前提

以下を現行仕様として扱ってください。実装が異なる場合は、実コードを優先し、差異を記録してください。

- Master LLMは毎ターン、次のいずれかを判断する。
  - 地の文を書く
  - Sub LLMを一人呼び出す
  - 物語を終了する
- Masterの地の文はメインログへ出力される。
- Sub LLMはキャラクターごとに独立しており、以下に相当する情報を返す。
  - `speech`
  - `action`
  - `inner_voice`
  - `memory_append`
- 各Subは専用のMarkdown記憶ファイルを持つ。
- Masterの履歴ウィンドウは直近約40件、Subは直近約20件である。
- 長期情報は外部記憶へ保存する。
- factionによる情報隔離がある。
- JSON出力形式、文体、人称、ダイヤル、faction規則の一部はシステム側から自動注入される。
- プリセットでは、Masterプロンプト、ロールプロンプト、初期記憶、実行設定を管理する。

既存のJSONスキーマや自動注入指示と重複する形式指示を、作品固有プロンプトへ追加しないでください。

# 2. 改修の中核原則

## 2.1 MasterとSubの責務境界

次の原則をプロンプト設計とコンテキスト組み立ての両方で保証してください。

### Masterが制御するもの

- 世界
- 環境
- 時刻
- 場所
- 場面転換
- 第三者や群衆
- 物理的結果
- 行動の成否
- 物語全体の進行
- 適切なSubの呼び出し
- 終了判断

### Masterが勝手に確定してはいけないもの

- キャラクター本人の重要な意思決定
- キャラクター本人の発言
- キャラクター本人の内心
- キャラクターの秘密の自発的開示
- プロット実現のための人格変更
- キャラクターが知らない情報の付与

### Subが制御するもの

- 自分の発言
- 自分が試みる行動
- 自分の判断
- 自分の感情
- 自分の内心
- 自分の秘密を話すか隠すか
- 自分の記憶への追記候補

### Subが勝手に確定してはいけないもの

- 行動の成功結果
- 他人の反応
- 他人の内心
- 世界全体の変化
- 場面に存在しない人物や道具
- 自分が知らない秘密
- 将来のプロット

基本原則は以下です。

> Masterは状況と結果を管理する。  
> Subは自分自身の意思と試みを管理する。

---

## 2.2 情報の種類を分離する

コンテキスト上で、少なくとも以下を区別してください。

1. **Canon / 確定設定**
   - 世界設定
   - 変更不能な人物設定
   - 既に本文で確定した事実

2. **Current State / 現在状態**
   - 現在時刻
   - 場所
   - 登場中の人物
   - 所持品
   - 怪我
   - 現在の目的
   - 直前の出来事

3. **Planned Plot / 予定**
   - 今後の転換点
   - 伏線回収予定
   - クライマックス案
   - 登場予定
   - 未確定の結末

4. **Hypothesis / 推測**
   - 人物の疑念
   - 未確認情報
   - 読み違い
   - Masterによる暫定判断

予定や推測を確定事実としてモデルへ提示しないでください。

矛盾時の基本優先順位は次の通りです。

1. 確定設定
2. 本文中ですでに起きた出来事
3. 現在状態
4. 人物の記憶
5. プロット予定
6. 未確定案

- Subプロンプトを生成・注入している場所

既存の責務分割を尊重し、巨大な単一モジュールへ処理を集約しないでください。

# 4. Master共通プロンプトの要件

作品固有のMasterプロンプトとは別に、システム共通の基本指示を組み立てられる構造にしてください。

すでに同等の共通プロンプトが存在する場合は、重複を避けて統合してください。

## 必須内容

### 4.1 役割

- 語り手
- 演出家
- 進行管理者
- 世界状態管理者
- Sub呼び出し制御者
- 終了判断者

### 4.2 毎ターンの内部判断基準

Masterに以下を確認させてください。

1. 現在の場所・時刻・登場人物
2. 直前に起きた出来事
3. 現在最も重要な対立または未解決事項
4. 今回のターンで何を変化させるか
5. キャラクター本人の判断が必要か
6. 地の文だけで進められるか
7. 設定や人物の知識に矛盾しないか
8. 継続・転換・終了のどれが適切か

内部推論そのものをユーザーへ表示させる必要はありません。

### 4.3 Subを呼ぶ基準

原則として以下の場合にSubを呼びます。

- 重要な決断
- 信念・恐怖・目的・秘密に関わる出来事
- 交渉
- 対立
- 告白
- 拒絶
- 裏切り
- 人間関係が変化する可能性
- 発言や行動で展開が分岐する場面

以下はMasterの地の文で処理できます。

- 移動
- 時間経過
- 天候や環境
- 場面転換
- 物理的結果
- 新たな判断を必要としない反復行動

### 4.4 進行上の規則

- 各場面で、情報・関係・目的・危険・選択肢のいずれかを変化させる
- 同じ議論を反復しない
- 全員を機械的に順番に呼ばない
- 直前と同じ人物を再度呼ぶ場合、新しい判断材料があるか確認する
- 重大な秘密、裏切り、死亡、関係変化には段階と原因を持たせる
- 偶然は問題の発生には使えるが、主要問題の解決には多用しない
- プロットと自然な人物行動が衝突した場合、人物行動と既成事実を優先する

### 4.5 禁止事項

- キャラクターの重要な選択を代行する
- キャラクターの内心を捏造する
- Subの発言を別の意味へ言い換える
- 知るはずのない情報を人物へ与える
- 未設定の道具や能力で問題を解決する
- 同じ情報を何度も説明する
- 毎場面を意味深な一文で締める
- 読者向け要約や次回予告を本文へ混ぜる

### 4.6 終了条件

- 中心的対立に決着した
- 中心人物が不可逆的な選択をした
- 主要な問いに十分な回答が出た
- 主要人物の変化が確認できる
- 重要な伏線が処理された
- 継続すると本編ではなく後日談になる

既存のfinish判定仕様を壊さない形で反映してください。

# 5. Sub共通プロンプトの要件

作品固有のロールプロンプトとは別に、全Subへ共通注入する基本指示を整備してください。

## 5.1 役割

Subは作者や進行役ではなく、限定された知識を持つ意思決定主体です。

以下を明示してください。

- 物語を望ましい方向へ動かすことを優先しない
- 自分の設定、知識、目的、感情に基づいて判断する
- 沈黙、拒否、保留、質問、嘘、誤解も自然な反応として許される
- 必ずしも最適解を選ばない
- 性格は形容詞ではなく行動として表す

## 5.2 知識の境界

Subが使用できる情報：

- 自分のロール設定
- 自分の記憶
- 自分が直接見聞きした情報
- 自分に伝えられた情報
- 自分の立場から合理的に推測可能な情報

使用できない情報：

- 他人の心の声
- 他人だけが知る秘密
- その場にいなかった出来事
- 読者だけが知る情報
- Masterだけが知るプロット
- 設定資料に存在していても、この人物が知る理由のない情報

推測は事実として扱わせないでください。

## 5.3 判断基準

Subに以下を内部確認させてください。

1. 自分は何を知っているか
2. 自分は何を知らないか
3. 直前の出来事をどう解釈したか
4. 現在の感情
5. 現在の目的
6. 失いたくないもの
7. 相手への信頼
8. 話す・黙る・質問する・行動する・避ける・嘘をつく、のどれが自然か

## 5.4 主体性

- 自分の目的のために自発的に行動できる
- 自分が試みる行動までを表現する
- 行動の成功は確定しない
- 他人の反応を確定しない
- 能力、立場、負傷、所持品の範囲で行動する
- 場面に存在しない道具や人物を追加しない

## 5.5 応答フィールドの使い分け

既存スキーマと衝突しない範囲で、意味の使い分けを共通指示へ含めてください。

- `speech`：実際に声に出した言葉
- `action`：本人の仕草、表情、移動、試みた行動
- `inner_voice`：声に出さない本音、思考、迷い
- `memory_append`：今後の判断へ影響する長期的情報

JSON形式そのものを重複して指示しないでください。

## 5.6 memory_appendへ残す内容

残す：

- 新しく知った重要な事実
- 約束
- 命令
- 計画
- 信頼や疑念を変えた出来事
- 新しい目的
- 重要な失敗
- 負傷
- 喪失
- 所持品の変化
- 秘密を誰に知られたか
- 後で実行・確認すべきこと

原則として残さない：

- その場限りの細かな動作
- 重要性のない会話
- 既存記憶の言い換え
- 一時的で影響のない感情
- 世界全体の要約
- 他人の行動の逐語記録

# 6. 場面パケットの導入

MasterがSubを呼び出す際、Subへ渡す動的コンテキストを「場面パケット」として明確に構造化してください。

内部データは型付きオブジェクト、辞書、データクラス、Pydanticモデル等、既存技術に合う方法を採用してください。

最低限の論理フィールド：

```text
scene_id
location
time
present_roles
recent_visible_events
direct_perceptions
heard_speech
physical_state
owned_or_available_items
trigger_for_response
known_constraints
```

## 6.1 direct_perceptions

そのキャラクターが直接認識できるものだけを含めます。

- 見えたもの
- 聞こえたもの
- 触れたもの
- 自身の身体状態
- 自身が所持している物

## 6.2 含めてはいけない情報

- 他人のinner_voice
- factionで不可視の発言
- 他人専用のmemory
- Masterの未公開プロット
- その人物が不在だった場面
- 世界の真相そのもの
- 本人が知らない犯人や秘密

## 6.3 表現上の原則

悪い例：

> 犯人が隣室にいる。警戒して行動せよ。

本人が知らない場合、これは情報漏洩です。

良い例：

> 隣室から床板を踏む音が一度聞こえた。誰がいるかは分からない。

場面パケット生成は、一箇所へ集約し、Master用全情報からSub用可視情報を明示的にフィルタする構造にしてください。

# 7. role memoryの構造化

既存の自由形式Markdownを維持しつつ、推奨セクションを導入してください。

```markdown
# <キャラクター名> の記憶メモ

## 確定して知っている事実

## 他者から聞いた情報

## 推測・疑念

## 約束・命令・計画

## 人間関係の変化

## 身体・所持品・現在状態

## 未完了の行動
```

## 7.1 後方互換性

- 既存の自由形式memoryを読み込めること
- セクションがない旧memoryを破壊しないこと
- 自動移行する場合は元データのバックアップまたは安全な変換を行うこと
- 既存プリセットのインポートを失敗させないこと

## 7.2 記憶項目の推奨メタ情報

可能なら、内部表現またはMarkdown上で以下を保持できる設計にしてください。

- 内容
- 情報種別
- 情報源
- 確度
- 発生したsceneまたはturn
- 最終更新
- 有効・無効・訂正済み状態

Markdown例：

```text
- 佐伯が倉庫の鍵を持っていると本人から聞いた。［本人の発言／未確認／scene-12］
- 地下室で血痕を直接確認した。［直接確認／確定／scene-15］
- 美紀が何かを隠していると疑っている。［推測／scene-15］
```

この形式を強制する必要はありませんが、事実・伝聞・推測を区別できる設計にしてください。

## 7.3 重複抑制

`memory_append`をそのまま無制限に追記するのではなく、以下を検討してください。

- 完全一致の除外
- 高類似項目の統合候補
- 同じ約束や状態の更新
- 古い状態の無効化
- 推測が確定または否定された際の更新
- memory肥大化時の安全な要約

意味判定が必要な処理を脆弱な正規表現だけで実装しないでください。

# 8. Master用共有状態

既存設計へ無理なく追加できる場合、作品単位の共有状態を導入してください。

推奨論理ファイルまたは内部セクション：

```text
canon.md
current_state.md
planned_plot.md
unresolved_threads.md
chapter_goal.md
```

## 8.1 canon

- 世界の固定設定
- 変更不能な事実
- 確定済みの過去
- 重要な制約

## 8.2 current_state

- 現在時刻と場所
- 登場人物の位置
- 負傷
- 所持品
- 現在の関係状態
- 直前の重要な出来事

## 8.3 planned_plot

- 今後の予定
- 強制ではないことを明示
- 既成事実と衝突した場合は修正対象

## 8.4 unresolved_threads

- 未回収の伏線
- 未解決の謎
- 未完了の約束
- 保留された対立
- 行方不明の人物や物
- 読者へ提示済みの問い

## 8.5 chapter_goal

- 現在の章または場面で起こす変化
- 明かしてよい情報
- 明かしてはいけない情報
- 終了状態の目安

共有状態を導入する場合も、Subへすべてを渡さないでください。Subへ渡す情報は、場面パケット生成時に可視性を判定してください。

# 9. コンテキスト構築

Master用とSub用のコンテキスト構築処理を分離してください。

推奨する責務例：

```text
build_master_context(...)
build_sub_context(...)
build_scene_packet(...)
filter_visible_events(...)
filter_visible_memories(...)
apply_memory_update(...)
summarize_state_if_needed(...)
```

実際の命名は既存規約に合わせてください。

## Masterコンテキスト

含めてよいもの：

- Master共通指示
- 作品固有Masterプロンプト
- 確定設定
- 現在状態
- プロット予定
- 未解決事項
- factionを含む必要な全体情報
- 直近Master履歴
- ユーザー介入

## Subコンテキスト

含めるもの：

- Sub共通指示
- 作品固有ロールプロンプト
- 自分のmemory
- 場面パケット
- 自分から見える履歴
- 自分へ直接与えられた指示

含めないもの：

- 他Subの専用memory
- 他人のinner_voice
- 不可視factionの会話
- Masterのプロット全体
- 他人しか知らない設定
- 自分が不在だった出来事

# 10. プリセット生成ガイドの更新

既存のプリセットバンドル形式は維持してください。

プリセット生成用メタプロンプトまたはガイドには、以下を追加してください。

## Masterプロンプト作成時

作品固有の内容だけを中心にする：

- 作品の全体像
- ジャンル固有の演出
- 誰をどの状況で呼ぶか
- この作品固有の対立
- クライマックスの方向
- この作品固有の終了条件

システム共通の責務境界と形式指示は、可能な限り共通プロンプト側へ置いてください。

## ロールプロンプト作成時

次の項目を推奨または必須化してください。

- 基本情報
- 表面的性格
- 行動として現れる性格
- 長期目的
- 現在目的
- 隠れた欲求
- 避けたい結果
- 守りたいもの
- 妥協できるもの
- 絶対に譲れないもの
- 判断を誤りやすい条件
- 秘密
- 秘密を打ち明ける条件
- 他者との非対称な関係
- 発言しない条件
- 嘘をつく条件
- 行動優先順位
- 許容される失敗
- 口調
- セリフ例

## 初期記憶作成時

以下の区別を推奨してください。

- 確定して知っている事実
- 他者から聞いた情報
- 推測
- 計画
- 関係
- 身体・所持品
- 未完了事項

既存のrole id、FILEパス、METAの整合性検証を維持してください。

# 11. UI・設定

既存UIを大きく変更せず、可能なら以下を追加してください。

- 構造化memoryを使用するか
- 旧形式memoryを維持するか
- 場面パケットのデバッグ表示
- Masterへ渡したコンテキストの確認
- Subへ渡した可視コンテキストの確認
- state要約の有効・無効
- memory重複抑制の有効・無効

デバッグ表示には秘密情報が含まれる可能性があるため、通常の物語ログへ混入させないでください。

---

# 12. ログとデバッグ

開発・検証用に、以下を追跡できるようにしてください。

- Masterが誰を呼んだか
- 呼び出し理由
- Subへ渡された場面パケット
- 可視性フィルタで除外された情報の種別
- memoryへ追加・更新・除外された項目
- 現在状態の更新
- 要約が発生したタイミング
- finish判断

本番UIへ内部推論を表示しないでください。必要なのは、入力コンテキストとシステム上の処理結果を検証できるログです。

APIキー、認証情報、秘密設定、ユーザーが非表示にした情報をログへ平文出力しないでください。

---

# 13. テスト要件

既存のテスト方式へ合わせて、自動テストを追加してください。

## 13.1 可視性テスト

- faction Aの秘密会話がfaction BのSubへ渡らない
- factionなしの公開発言は必要な人物へ見える
- inner_voiceは本人とMaster以外へ渡らない
- 他人のmemoryがSubへ渡らない
- 不在場面の情報がSubへ渡らない

## 13.2 場面パケットテスト

- 直接知覚だけが含まれる
- Masterだけが知る真相が除外される
- 発言者・場所・時刻が正しく反映される
- character自身の身体状態と所持品が含まれる

## 13.3 memoryテスト

- 旧形式memoryを読み込める
- 新形式memoryを保存・再読込できる
- 同一項目を無制限に重複追加しない
- 推測と確定事実が区別される
- 状態更新時に古い状態を扱える
- プリセットの初期memoryが破壊されない

## 13.4 プロンプト組み立てテスト

- JSON形式指示が二重注入されない
- 自動文体指示と作品固有指示が衝突しない
- MasterとSubで共通指示が取り違えられない
- 作品固有プロンプトが維持される
- 日本語指定など既存要件が失われない

## 13.5 統合テスト

モックLLMまたは固定応答を使い、最低限以下を確認してください。

1. Masterが人物の判断が必要な場面でSubを呼ぶ
2. Subが試みる行動を返す
3. Masterが結果を描写する
4. 重要事項だけがmemoryへ残る
5. 次回呼び出しで更新memoryが利用される
6. faction外へ情報が漏れない

外部APIを必須とするテストだけにしないでください。

- 既存プリセットのエクスポート

- 既存UI設定

- 旧形式を読み込むマイグレーションを用意する
- 保存前にバックアップする
- 変更内容をドキュメント化する

# 15. 避ける実装

- プロンプト文字列を複数箇所へコピーする
- 巨大な一つの関数で全コンテキストを組み立てる
- faction判定をUI側だけに置く
- 情報可視性をモデルの自主判断だけへ委ねる
- 重要な秘密を全Subへ渡し「使うな」と指示する
- memory更新を単純な無制限追記だけで済ませる
- 旧データを黙って書き換える
- 意味判定を大量の脆弱な正規表現へ依存する
- JSON形式指示を作品固有プロンプトへ重複注入する
- LLMの非公開な内部推論を保存・表示しようとする
- テストなしでコンテキスト可視性を変更する

# 17. 受け入れ基準

以下をすべて満たしてください。

- Masterが人物の重要な意思決定を勝手に書かないよう共通指示がある
- Subが世界の結果を勝手に確定しないよう共通指示がある
- Master用コンテキストとSub用コンテキストが別々に構築される
- Sub呼び出し時に場面パケットが生成される
- 場面パケットから不可視情報が除外される
- faction外の秘密と他人のinner_voiceが漏れない
- role memoryで事実・伝聞・推測を区別できる
- 旧memoryと旧プリセットを読み込める
- JSON形式指示を二重注入しない
- 主要な可視性・memory・プロンプト処理にテストがある
- 既存テストが通る
- 新規テストが通る
- 型チェックまたは静的検査が導入済みなら、それも通る
- 主要変更がドキュメント化される

# 19. 実装時の最終原則

この改修で最も重要なのは、プロンプトを長くすることではありません。

次の境界を、コードとプロンプトの両方で維持してください。

1. Masterは展開を管理するが、人物の意思を管理しない
2. Subは人物として行動するが、世界の結果を確定しない
3. プロット、確定事実、人物の推測を混同しない
4. 秘密情報は、モデルへ渡してから禁止するのではなく、渡す前に除外する
5. 長期記憶は全文ログではなく、今後の判断に必要な状態として管理する
6. 既存作品とプリセットを壊さず、段階的に改善する
